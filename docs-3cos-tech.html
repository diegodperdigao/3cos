1. Visão geral da arquitetura

Stack

  • Frontend: HTML + CSS + Vanilla JavaScript, sem build system. Scripts carregados via <script src> com cache busting (?v=144).
  • Estado global: objeto STATE mantido em memória + persistência via localStorage (saveToLocal()) e Supabase (camada js/data.js).
  • Ícones: Lucide via data-lucide="..." + lucide.createIcons() após cada render.
  • Temas: combinação data-edition + data-theme no <html>. Editions: default, mono, bento, meridian, neonflow, glass. Modes: dark, light.
  • Backend IA: /api/ai serverless Vercel → Google Gemini (free tier).
  • Banco: Supabase (Postgres) — schema em supabase/schema.sql.

Fluxo de boot

  1. users.js carrega último usuário do localStorage / Supabase.
  2. Se há sessão → chama showHub(). Senão → renderiza tela de lock.
  3. showHub() (em app.js) faz fade out do lock, monta greeting, chama buildHubCards(), buildHubWidgets(), renderHolidayPostIt(), updateCopilotVisibility(), applyAppTheme(), initMosaics(), lucide.createIcons().
  4. Após 3s, roda runPaymentWatchdog() silenciosamente (alertas vão pro sino, não pra toast).

Convenções globais

  • Funções globais: tudo exposto via window.fnName = ... para acesso entre arquivos sem imports.
  • Helpers de formatação: fc(v) = currency BRL, pct(a,b) = porcentagem, medal(i) = badge de ranking.
  • Avatar: userAvatar(nameOrUser, size) retorna <img> se tiver URL, senão círculo com iniciais coloridas (cor determinística do nome).
  • Modal: openModal(title, bodyHTML, footerHTML) + closeModal().
  • Toast: toast(text, type) — type: 's' success, 'w' warning, 'e' error.

Como os 3 sistemas se conectam

Hub é o container visual: greeting + faixa de widgets + grid de módulos.
Widgets são montados pelo hub via buildHubWidgets() e leem STATE diretamente.
Copilot é independente (FAB flutuante visível em qualquer tela quando STATE.betaMode === true) — recebe STATE resumido como contexto a cada mensagem.

2.4. Renderização — showHub()

  1. Fade out lock screen (opacity: 0 → display: none após 650ms).
  2. Pega primeiro nome: STATE.user.name.split(' ')[0].
  3. Preenche header: #hub-uname, #hub-urole (usa title > ROLES[role].label > role).
  4. Avatar via userAvatar(STATE.user, 32).
  5. Greeting: Bem-vindo(a), <strong>{firstName}</strong> — selecione o módulo de trabalho.
  6. Fade in hub (display: flex → opacity: 1 após 50ms).
  7. Chama em sequência: buildHubCards(), buildMobileHome(), updateNotifBadge(), buildHubWidgets() (se existir), updateLabButton(), syncBetaAttributes(), updateCopilotVisibility(), applyAppTheme(), initMosaics(), lucide.createIcons().
  8. Após 3s, dispara runPaymentWatchdog() silenciosamente (alertas vão para o sino).

2.5. Renderização do grid — buildHubCards()

function buildHubCards(){
  const userMods = STATE.user?.modules || [];
  const isAdmin = STATE.user?.role === 'admin';
  // Admin vê tudo; user comum só vê módulos atribuídos a ele
  const visible = MODS
    .filter(m => !m.adminOnly || isAdmin)
    .filter(m => isAdmin || userMods.includes(m.id));
  document.getElementById('hub-cards').innerHTML = visible.map(m => `
    <div class="hub-app" onclick="openMod('${m.id}')"
      style="--app-border:${m.color};--app-glow:${m.glow};--app-bg:${m.bg};--app-stroke:${m.stroke}">
      <div class="hub-app-icon"><i data-lucide="${m.icon}"></i></div>
      <div class="hub-app-name">${m.label}</div>
    </div>`).join('');
  // Auto-ajusta colunas (max 6) baseado em quantos módulos o usuário tem acesso
  const count = visible.length;
  document.getElementById('hub-cards').style.gridTemplateColumns = `repeat(${Math.min(count,6)},1fr)`;
  lucide.createIcons();
}
Variáveis CSS inline: cada card recebe 4 custom properties (--app-border, --app-glow, --app-bg, --app-stroke) que o CSS consome para colorir borda/sombra/fundo/ícone — permite cor diferente por módulo sem CSS específico para cada um.

2.6. Navegação entre hub e módulos

openMod(id): abre um módulo a partir do hub (ou troca entre módulos).

  1. Salva id em sessionStorage['3cos_activeMod'] (para restaurar após F5).
  2. Se já há módulo ativo: fade out (320ms) → display: none → buildMod(id, el) → fade in.
  3. Se está vindo do hub: fade out hub → buildMod() → fade in módulo.
  4. buildMod(id, el) está em try/catch — se quebrar, mostra fallback "Erro ao abrir o módulo" + botão "Voltar ao Hub".
  5. Atualiza bottom nav mobile via updateBottomNav(id).

goBack(): volta ao hub.

  1. Limpa sessionStorage['3cos_activeMod'].
  2. Fade out de todos os .mod.active.
  3. Fade in do #hub.
  4. Destrói chart instances (window.mainChartInstance.destroy()) para evitar memory leak.
  5. Reconstrói: buildHubCards() · buildMobileHome() · updateNotifBadge() · initMosaics() · lucide.createIcons().

Renderizada quando a conversa ativa não tem mensagens. Mostra ícone Gemini SVG + título + 4 chips de sugestão clicáveis:

3.6. Fluxo de envio de mensagem (sendCopilotMessage)

  1. Lê texto do #copilot-input. Aborta se vazio ou _copilotSending já ativo.
  2. Pega ou cria conversa ativa.
  3. Auto re-sync: se STATE.affiliates e STATE.payments estão vazios, chama Data.loadAll() antes de enviar (evita "tenho dados stale" após fresh login).
  4. Loga estatísticas do estado: { hasUser, affiliates, payments, contracts, tasks, brands, reports }.
  5. Push da mensagem do usuário em conv.messages. Se for a primeira, gera conv.title via _titleFromText (max 40 chars).
  6. Limpa input, marca _copilotSending = true, salva e re-renderiza (mostra typing dots).
  7. Constrói contexto com buildCopilotContext().
  8. POST /api/ai com { messages: conv.messages, context }.
  9. Se OK → push da reply do assistant. Se erro → _friendlyCopilotError(err) mostra mensagem amigável em PT.
  10. Marca _copilotSending = false, atualiza conv.updatedAt, salva, re-renderiza.

3.7. Contexto enviado à IA — buildCopilotContext()

Otimizado para minimizar tokens. Estrutura:

{
  today: "2026-04-28",
  user: { name, role },
  brands: { ...todas as marcas... },

  // AFILIADOS: só ativos, campos compactos, notas truncadas em 100 chars
  affiliates: [{
    id, name, contractType, ftds, qftds,
    deposits, netRev, commission, profit, deals,
    notes?, tags?
  }],

  contracts: [{ affiliate, brand, type, value, paid, paymentStatus }],

  // PAGAMENTOS: agregado por status + lista somente dos críticos
  payments: {
    by_status: { pendente: { count, total }, vencido: {...}, ... },
    critical: [...só vencidos+atrasados, item a item...]
  },

  tasks_open: [{ title, priority, status, assignee, dueDate?, description? }],

  closings: [...últimos 6...],

  // REPORTS em DOIS formatos:
  reports_monthly: [...agregado afiliado×marca×mês — economia de tokens...],
  reports_daily_last30: [...linhas diárias dos últimos 30 dias para perguntas precisas...],

Truque de tokens: agregar reports por mês economiza muito (1 linha por mês vs 30 por mês). O reports_daily_last30 mantém detalhe recente para perguntas como "ontem" ou "dia 3 de abril".

3.8. Backend — api/ai.js (Vercel serverless)

Configuração

  • Endpoint: POST /api/ai
  • Env vars: GEMINI_API_KEY (obrigatório, pega em aistudio.google.com/apikey), GEMINI_MODEL (opcional)
  • Modelo padrão: gemini-1.5-flash
  • Fallback chain: se o modelo retornar quota exceeded, tenta na ordem 1.5-flash → 1.5-flash-8b → 2.5-flash-lite → 2.0-flash
  • maxOutputTokens: 2048 · temperature: 0.4

System prompt (resumo)

  • Identidade blindada: "Você é o 3C Copilot. Nunca mencione Google, Gemini, LLM, modelo, treinamento. Se perguntarem 'quem é você', responda sempre: 3C Copilot, criado pela 3C Gaming, exclusivamente para o 3C OS Pro."
  • Contexto de negócio (iGaming): 3C = operadora, afiliados trazem jogadores, comissões CPA/Rev Share/híbrido, lucro 3C = receita marca − comissão paga.
  • Glossário: FTD, QFTD, Net Revenue, CPA, Rev Share.
  • Regras de resposta: PT-BR, conciso, listas, markdown. Não inventar dados — usar SEMPRE o JSON snapshot. Apenas consultar (não pode executar ações).
  • Para perguntas sobre dia específico → reports_daily_last30. Totais históricos → reports_monthly.

Truque do "primed context" (importante)

O backend NÃO envia o contexto como systemInstruction (que o Gemini às vezes ignora). Em vez disso, fabrica uma falsa primeira troca user→model onde:

  1. O "user" entrega o JSON do snapshot.
  2. O "model" finge confirmar listando contagens reais (afiliados, top 3 por lucro, marcas, contratos, pagamentos por status, tarefas).

Como o modelo "já disse" esses números, ele os trata como verdade conhecida nas perguntas seguintes. Fica muito mais difícil dele alucinar ou dizer "não tenho acesso aos dados". Implementado em _buildAckMessage(ctx).

3.9. Tratamento de erros

_friendlyCopilotError(err): traduz erros técnicos em PT amigável:

  • 429 / quota / limit: "Copilot está com muitas solicitações... aguarde ~Xs e tente de novo." Extrai retry hint do erro raw.
  • API key missing: "Copilot ainda não foi configurado por um administrador."
  • Network / timeout / status 0: "Não consegui me conectar... verifique sua conexão."
  • 5xx: "Copilot está temporariamente indisponível."
  • Default: "Tive um problema para processar sua pergunta. Tente reformular."

_sanitizeCopilotMessage(content): roda em RENDER time. Se o texto da resposta tem padrões de erro raw da API (ex: quota exceeded, generativelanguage.googleapis.com, retry in 30s, resource_exhausted) → substitui pelo texto amigável. Protege mensagens antigas no localStorage de antes da implementação dos friendly errors.

3.10. Renderização — pontos importantes

  • renderCopilot(): monta corpo (welcome ou lista de mensagens). Adiciona indicador cp-typing com 3 dots se _copilotSending. Roda lucide.createIcons() e re-renderiza histórico.
  • renderCopilotMessage(m): bubble do user (sem avatar) ou assistant (com ícone Gemini). Aplica _sanitizeCopilotMessage + formatMarkdown em mensagens do assistant.
  • formatMarkdown(text): regex simples para **bold**, *italic*, `code`, listas - item, quebras de linha. Faz escape de &<> primeiro.
  • renderCopilotHistory(): lista ordenada por updatedAt desc. Mostra título, timestamp relativo (formatRelativeTime: "agora", "há 5m", "há 2h", "há 3d", data), número de mensagens. Botão lixeira por item. Toolbar de seleção bulk se _copilotSelectMode.
  • onCopilotKeydown(e): Enter envia, Shift+Enter quebra linha.

3.11. Visibility safety net

updateCopilotVisibility() mostra FAB se STATE.betaMode === true && STATE.user. Como STATE pode carregar async (Supabase), há um intervalo no DOMContentLoaded que re-checa a cada 2s nos primeiros 20s para evitar FAB invisível em sessões com login restaurado.

function _activeWidgets() {

4.4. Renderers individuais

_widgetNotifications

  • Lê STATE.notifications, pega as 4 primeiras.
  • Cada item: dot colorido por n.type (red/amber/green/theme) + texto + tempo.
  • Vazio: "Nada novo por aqui."
  • Footer: "Ver todas" → toggleActionCenter().

_widgetPaymentsQueue

  • Agrupa STATE.payments por status via computePaymentStatus(p) (que considera vencimento real, não só campo status).
  • Mostra 4 stat rows: Vencidos (red) · Em atraso (amber) · Pendentes (text2) · Pagos (mês) (green). Cada um com count + total formatado em BRL.
  • Footer: "Abrir financeiro" → openMod('payments').

_widgetTasks

  • Filtra STATE.tasks não concluídas, pega as 4 primeiras.
  • Cada item: dot priorizado (.hw-pri-a/m/b) + título + meta (assignee + dueDate em pt-BR).
  • Vazio: "Tudo em dia. 🎯" + footer "Nova tarefa".
  • Cheio: footer "Abrir todas (N)".

_widgetResults

  • Calcula KPIs do mês corrente a partir de STATE.reports filtrados por monthKey (YYYY-MM).
  • Soma: ftd, qftd (suporta tanto número quanto objeto {brand: count}), netRev.
  • Fallback: se nenhum report no mês, faz rollup de todos os afiliados (a.ftds + a.qftds + a.netRev) e troca o título para "Resultado consolidado" com label "Total acumulado".
  • Body: .hw-results-big com QFTD em destaque + 2 stat rows (FTDs, Net Revenue).
  • Footer: "Ver dashboard" → openMod('dashboard').

_widgetTopAffiliates

  • Filtra afiliados com profit > 0, ordena desc, top 5.
  • Cada item: rank (1-5) + nome + lucro formatado em BRL.
  • Título: "Top afiliados · Lucro 3C". Footer: "Ver todos" → openMod('affiliates').

_widgetPipelineStatus

  • Lê STATE.pipeline.stages e STATE.pipeline.cards.
  • Para cada stage, conta cards com stageId === s.id. Mostra dot colorido (s.color) + nome + count.
  • Vazio: "Nenhuma negociação no funil." + footer ainda abre o módulo.

_widgetRecentActivity

  • Lê STATE.auditLog, pega os 4 mais recentes.
  • Cada item: ação (l.action) + meta (l.user || 'Sistema' · l.time).
  • Footer: "Ver auditoria" → openMod('audit').

4.5. Mounting — buildHubWidgets()

window.buildHubWidgets = () => {
  const wrap = document.getElementById('hub-widgets');
  if (!wrap) return;
  const active = _activeWidgets();
  if (!active.length) { wrap.innerHTML = ''; return; }

  const renderMap = {
    notifications: _widgetNotifications,
    payments_queue: _widgetPaymentsQueue,
    tasks: _widgetTasks,
    results: _widgetResults,
    top_affiliates: _widgetTopAffiliates,
    pipeline_status: _widgetPipelineStatus,
    recent_activity: _widgetRecentActivity,
  };
  wrap.innerHTML = active.map(id => {
    const fn = renderMap[id];
    return fn ? fn() : '';
  }).join('');
  if (typeof lucide !== 'undefined') lucide.createIcons();
};

Chamado por showHub() após login. A ordem dos widgets na renderização segue a ordem do array salvo em STATE.settings.hubWidgets.

4.6. Picker modal — openHubWidgetPicker()

Acionado pelo botão "Personalizar widgets" no hub (lado direito do label "Módulos"). Abre openModal() com:

  • Nota explicativa: "Escolha quais widgets aparecem... A ordem de exibição segue a ordem marcada abaixo."
  • Lista de checkboxes (.hwp-row): cada item com checkbox + ícone + nome + descrição. checked determinado pelo _activeWidgets() atual.
  • Footer: Cancelar + Salvar.

4.7. Persistência — _saveHubWidgets()

window._saveHubWidgets = () => {
  const selected = [...document.querySelectorAll('.hwp-check:checked')].map(c => c.value);
  if (!STATE.settings) STATE.settings = {};
  STATE.settings.hubWidgets = selected;
  saveToLocal();        // localStorage
  closeModal();
  buildHubWidgets();    // re-render imediato
  toast('Widgets atualizados', 's');
};

Storage: ID dos widgets ativos vai para STATE.settings.hubWidgets, persistido em localStorage via saveToLocal() + sincronizado com Supabase pela camada data.js.

4.9. Como adicionar um novo widget

  1. Adicione entrada no array HUB_WIDGETS (id único, name, icon, desc).
  2. Crie a função renderer _widgetSeuNome() seguindo o padrão (lê STATE, retorna via _widgetShell()).
  3. Adicione mapeamento em renderMap dentro de buildHubWidgets().
  4. Opcional: adicione o id em DEFAULT_HUB_WIDGETS se quiser que apareça por padrão.
